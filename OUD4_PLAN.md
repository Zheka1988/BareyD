# План подготовки к ОУД4 (СТ РК ISO/IEC 15408-3-2017)

ОУД4 — "методически спроектирован, протестирован и проверен".

---

## 1. Задание по безопасности (ASE)

Главный документ — без него оценка не начнётся.

| Компонент | Что подготовить |
|-----------|----------------|
| **ASE_INT.1** | Введение ЗБ: название ОО, версия, общее описание назначения модуля |
| **ASE_SPD.1** | Определение проблемы безопасности: угрозы, политики, предположения |
| **ASE_OBJ.2** | Цели безопасности для ОО и среды функционирования |
| **ASE_REQ.2** | Функциональные требования (SFR) и требования доверия (SAR) — берутся из ISO 15408-2 |
| **ASE_TSS.1** | Краткая спецификация ОО — как функции модуля реализуют SFR |
| **ASE_CCL.1** | Утверждения о соответствии: какой профиль защиты, какой пакет |
| **ASE_ECD.1** | Определение расширенных компонентов (если используются не из каталога) |

**Итого:** один документ ~40-80 страниц.

---

## 2. Разработка (ADV)

Техническая документация на архитектуру и код.

| Компонент | Что подготовить |
|-----------|----------------|
| **ADV_ARC.1** | Описание архитектуры безопасности: как ОО защищает себя от вмешательства, обхода функций безопасности |
| **ADV_FSP.4** | **Полная** функциональная спецификация: все внешние интерфейсы (API, формы, кнопки) с описанием параметров, действий, ошибок, связи с SFR |
| **ADV_TDS.3** | Проект верхнего уровня с модульной декомпозицией: Django apps, модели, views, JS-модули, взаимодействие между ними |
| **ADV_IMP.1** | Представление реализации — доступ к **исходному коду** с пояснениями к критическим участкам (аутентификация, фильтрация, валидация) |

**Что делать прямо сейчас в коде:**
- Добавить комментарии к security-critical функциям (login, `@login_required`, `clean()`, фильтрация по правам)
- Задокументировать все API-эндпоинты (входы, выходы, ошибки)
- Нарисовать диаграмму архитектуры (компоненты, потоки данных)

|-----------|----------------|

## 4. Поддержка жизненного цикла (ALC)

Самый трудоёмкий блок — процессы разработки.

| Компонент | Что подготовить |
|-----------|----------------|
| **ALC_CMC.4** | Управление конфигурацией с **автоматизацией**: git, CI/CD, автоматическая сборка, контроль версий всех артефактов |
| **ALC_CMS.4** | Список конфигурации: все файлы ОО, документация, тесты — каждый элемент отслеживается |
| **ALC_DEL.1** | Процедура поставки: как передаётся ОО заказчику, контроль целостности (хэши, подпись) |
| **ALC_DVS.1** | Меры безопасности разработки: физическая/логическая защита среды разработки, контроль доступа к коду |
| **ALC_LCD.1** | Модель жизненного цикла: описание процесса разработки (этапы, роли, инструменты) |
| **ALC_TAT.1** | Инструменты разработки: Python 3.11, Django 5.2, PostgreSQL/PostGIS, список всех зависимостей с версиями |

**Что делать прямо сейчас:**
- Настроить CI/CD (GitHub Actions): линтинг, тесты, сборка
- Вести changelog
- Зафиксировать все версии инструментов
- Описать процесс разработки (git flow, code review, кто имеет доступ)

## 5. Тестирование (ATE)

| Компонент | Что подготовить |
|-----------|----------------|
| **ATE_FUN.1** | Функциональные тесты: каждый SFR должен быть покрыт тестом |
| **ATE_COV.2** | Анализ покрытия: матрица "SFR -> тест -> интерфейс" |
| **ATE_DPT.1** | Тестирование на уровне подсистем (не только чёрный ящик) |
| **ATE_IND.2** | Независимое тестирование — проводит **лаборатория**, но ты предоставляешь стенд |

**Что делать прямо сейчас:**
- Написать тесты (pytest/Django TestCase) на:
  - Аутентификацию (login, logout, доступ без авторизации)
  - API-эндпоинты (фильтры, маркеры, поиск)
  - Валидацию модели Object (геометрия, страна)
  - Права доступа (если появятся роли)
- Подготовить тестовый стенд для лаборатории

## 6. Анализ уязвимостей (AVA)

| Компонент | Что подготовить |
|-----------|----------------|
| **AVA_VAN.3** | Сфокусированный анализ уязвимостей: SQL-инъекции, XSS, CSRF, доступ без авторизации, обход фильтров |

**Что делать прямо сейчас:**
- Провести OWASP-проверку
- Убедиться что все view защищены `@login_required`
- CSRF-токены на всех формах
- Параметризованные запросы (Django ORM)
- Валидация входных данных (`_parse_ids`, `clean()`)
- Добавить rate limiting на API
- Настроить HTTPS, Secure cookies, HSTS
